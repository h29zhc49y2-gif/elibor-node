## 三、技术方案评估

### 3.1 当前问题

Hostinger 共享主机存在以下限制：
1. `.htaccess` 重写规则不生效
2. 无法自定义 Apache 配置
3. SSH 需要密码认证，无法自动化
4. Hostinger API 不包含数据库管理功能

### 3.2 引擎自动运算需求

聚栖项目有复杂的引擎逻辑需要考虑：
- **时间引擎**：6分钟 = 星球1小时，144分钟 = 星球1天
- **生活引擎**：栖人行动、消耗、产出
- **经济系统**：资源产出、建造
- **每日播报**：自动生成新闻
- **定时任务**：定期执行引擎逻辑

### 3.3 推荐方案

| 方案 | 后端 | 数据库 | 定时任务 | 免费额度 | 适合度 |
|------|------|--------|----------|----------|--------|
| **① Railway** | Node.js/Python | PostgreSQL | ✅ Cron | 5刀/月 | ⭐⭐⭐⭐⭐ |
| **② Render** | Node.js/Python | PostgreSQL | ✅ Cron | 750小时/月 | ⭐⭐⭐⭐ |
| **③ Fly.io** | Docker | SQLite/PostgreSQL | ✅ Cron | 3个免费实例 | ⭐⭐⭐⭐ |
| **④ Cloudflare Workers + D1** | Workers | D1 (SQLite) | ✅ Cron Triggers | 10万次/天 | ⭐⭐⭐ |

**最终推荐：Railway + PostgreSQL**

理由：
1. 支持定时任务 — 可运行引擎自动运算
2. 免费额度够用 — 5刀/月
3. 部署简单 — 连接 GitHub 自动部署
4. 自带 PostgreSQL — 和 MySQL 几乎一样
5. 支持 WebSocket — 后续做实时推送

## 七、技术栈调研（晚间）

### 7.1 调研背景

用户指出聚栖的核心特点是**全自动算法驱动的模拟养成系统**，与普通用户交互型游戏不同：
- 引擎必须 **24小时常驻运行**
- 每 **6分钟** 执行一次行为决策
- 栖人状态持续更新
- 用户只是"观察者"，核心玩法是算法自动产生

### 7.2 参考案例

#### Mochia（现代版 Neopets）
- 后端：Rust（axum + tokio + sqlx）
- 前端：SolidJS（仅 3 个依赖）
- 数据库：PostgreSQL
- 通信：WebSocket（二进制协议）
- 服务器：单台 $5/月 VPS
- 运行 3 年零崩溃、零数据丢失

#### 休闲游戏平台（国内主流）
- 前端：Vue.js 3 + TypeScript + PixiJS
- 后端：Node.js (TypeScript) / Python (FastAPI)
- 数据库：MySQL 8.0
- 缓存：Redis

### 7.3 聚栖特殊需求分析

| 需求 | 说明 | 技术要求 |
|------|------|----------|
| **常驻进程** | 引擎必须24小时运行 | 需要长期运行的后台服务 |
| **定时调度** | 每6分钟执行一次 | Cron Jobs 或 持续循环 |
| **状态持久化** | 栖人状态随时变化 | 数据库事务保证 |
| **并发处理** | 可能有1000+栖人同时运算 | 多线程/异步处理 |
| **实时推送** | 用户打开时看到最新状态 | WebSocket 或 轮询 |

### 7.4 方案评估

| 方案 | 适合场景 | 成本 | 推荐度 |
|------|----------|------|--------|
| **Railway + Node.js + PostgreSQL** | 独立开发者/快速上线 | $0-5/月 | ⭐⭐⭐⭐⭐ |
| **Fly.io + Docker + PostgreSQL** | 需要定时任务/灵活控制 | 免费起 | ⭐⭐⭐⭐ |
| **Rust + SolidJS + VPS** | 追求极致性能 | $5/月 | ⭐⭐⭐ |
| **Cloudflare 全家桶** | 全球化用户 | 免费 | ⭐⭐⭐ |

**不适合的方案**：
- Serverless (Vercel Functions)：函数有执行时间限制，无法运行持续循环
- Edge Functions：无状态，不适合维护栖人状态

**选择方案：Railway + Node.js + PostgreSQL**

理由：
