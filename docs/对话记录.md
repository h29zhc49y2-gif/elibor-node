## 📻 每日播报系统设计

### 一、核心概念

**"每日播报"**：用户首次登录时自动弹出，展示星球上过去"一天"（现实时间2.4小时）发生的大事和用户栖人的活动记录。

**时间换算**：

- 6分钟 = 星球1小时
- 144分钟（2.4小时）= 星球1天
- 显示格式："第127天"

### 五、技术实现方案

#### 5.1 实时更新机制

```
方案A：WebSocket实时推送（推荐）
- 服务器主动推送新动态
- 客户端实时接收并显示
- 优点：实时性强，资源消耗低
- 缺点：需要WebSocket支持

方案B：轮询（备用）
- 客户端定时请求新动态
- 频率：每10秒一次
- 优点：兼容性好
- 缺点：资源消耗较高
```

#### 5.2 数据结构

```sql
CREATE TABLE dynamic_events (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    type VARCHAR(20),      -- 动态类型
    icon VARCHAR(10),      -- 图标
    color VARCHAR(20),     -- 颜色
    soul_id VARCHAR(36),   -- 栖人ID
    soul_name VARCHAR(50), -- 栖人名字
    soul_level VARCHAR(20),-- 用户等级
    content TEXT,          -- 动态内容
    impact JSON,           -- 影响数据
    timestamp TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    is_user_soul BOOLEAN DEFAULT FALSE -- 是否是用户自己的栖人
);
```

## ⚖️ 经济系统方案对比

### 方案A：用户可操作经济系统

**核心逻辑**：玩家直接管理资源和货币

| 优点         | 缺点          |
| ---------- | ----------- |
| ✅ 玩家有直接控制权 | ❌ 可能过于复杂    |
| ✅ 增加策略深度   | ❌ 破坏"观察者"定位 |
| ✅ 增强参与感    | ❌ 栖人自主性降低   |

### 方案B：栖人自主经济系统

**核心逻辑**：栖人自己管理经济活动

| 优点          | 缺点         |
| ----------- | ---------- |
| ✅ 符合"观察者"定位 | ❌ 玩家参与感不足  |
| ✅ 栖人更有自主性   | ❌ 可能缺乏策略性  |
| ✅ 简化玩家操作    | ❌ 经济系统存在感低 |

---

### 方案C：混合经济系统（推荐）

**核心逻辑**：栖人自主运作 + 玩家有限干预
